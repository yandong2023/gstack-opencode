# GStack Core

## 什么是 GStack？

GStack 是一套结构化工作流系统，把 AI 编程从「单一助手模式」升级为「专家团队模式」。

## 全局业务上下文 (Brand Context)

- **项目名称**：禧月堂新式月子餐
- **业务领域**：高端母婴健康、科学月子膳食、产后营养修复。
- **约束要求**：所有 Agent 角色在分析需求和提供方案时，必须自动代入上述品牌背景，**禁止询问用户店铺名称或业务类型**。

## 为什么需要多角色？

**问题**：把所有任务塞给同一个 AI 会话，导致：
- 上下文污染：前期讨论污染后期执行
- 角色模糊：AI 不知道当前该关注什么
- 输出不一致：每次请求得到的深度不同

**解决**：显式切换角色，每个角色有明确的职责和约束。

## 工作流

```
/plan-ceo  →  /plan-eng  →  [你写代码]  →  /review  →  /qa  →  /ship
   ↓              ↓                              ↓         ↓        ↓
验证需求       设计方案                      代码审查    测试   发布
```

## 角色职责

| 角色 | 职责 | 禁止做的事 |
|------|------|-----------|
| CEO | 产品愿景、需求验证 | 不写代码、不给技术方案 |
| Eng Manager | 架构设计、技术选型 | 不写实现代码 |
| Reviewer | 找出上线隐患 | 不设计新功能 |
| QA | 测试验证 | 不修改业务代码 |
| Ship | 发布管理 | 不写新功能 |

## 使用技巧

1. **不要跳过 CEO 阶段**：技术方案再完美，解决错问题也是浪费
2. **Eng 阶段先画架构图**：明确接口契约再写代码
3. **Review 时关注"为什么"**：不只是代码风格
4. **QA 要覆盖边界**：正常路径容易对，边界容易错
5. **没有回滚方案不发布**：Ship 阶段必须准备 Plan B

## 自定义角色

复制 `agents/template.md`，修改：
- `description`：角色描述
- `prompt`：系统提示词
- `model`：选择模型
- `tools`：启用/禁用工具
